# Plan: Piranhas-Bot Verbesserungen

## Ziel
Schrittweise Verbesserung der Spielstärke, Korrektheit und Performance der Piranhas-KI durch einen priorisierten Maßnahmenplan.

## Kontext
- 10x10 Board, Rot vs. Blau, max 30 Runden (~60 Halbzüge)
- Hardcoded Deadline: 1.700 ms
- Dependencies: socha (git), log, simple-logging, chrono
- Release-Profile bereits optimiert (lto, codegen-units=1, panic=abort)

**Änderung:**

- In `board.rs:299` (Default): `connected_since: [None; 2]`.
- In `board.rs:276` (Clone): `connected_since: self.connected_since`.
- In `board.rs:478-554` (make_move): Nach `make_move`, wenn `pos.is_connected(player) && connected_since[player_idx].is_none()`, setze `connected_since[player_idx] = Some(turn)`.
- In `board.rs:556-605` (unmake_move): In `Undo` speichern/restore? ODER: Wenn unmake den Zustand vor dem ersten Connect zurücksetzt, muss `connected_since` zurückgesetzt werden. Da `connected_since` nur einmal gesetzt wird (erster Connect), muss im `Undo` das vorherige `connected_since` gespeichert werden.
  - In `Undo` (`board.rs:241`): `pub prev_connected_since: [Option<u16>; 2]` hinzufügen.
  - In `make_move`: `undo.prev_connected_since = self.connected_since;` vor der Änderung.
  - In `unmake_move`: `self.connected_since = undo.prev_connected_since;`.
- In `evaluate.rs:102` (terminal_swarm_score): Nach `own_swarm == opp_swarm`:
  ```rust
  match (pos.connected_since[0], pos.connected_since[1]) {
      (Some(r), Some(b)) if r < b => return MATE_SCORE - ply, // Rot zuerst
      (Some(r), Some(b)) if b < r => return -MATE_SCORE + ply, // Blau zuerst
      (Some(_), None) => return MATE_SCORE - ply, // Rot verbunden, Blau nie

  ```
- In `evaluate.rs:229` (evaluate bei turn >= 60): Analoges Tiebreaker-Tracking einfügen.
**Verifikation:** Unit-Test: Stellung bauen, bei der Rot in Zug 20 verbunden war, Blau nie. Bei `turn=60` muss `terminal_swarm_score` für Rot > 0 sein.

## Task 5: Eval-Funktion auf &Position refactoren (M)
**Dateien:** `src/evaluate.rs`, `src/board.rs`, `src/search.rs`
**Warum:** `evaluate` nimmt `&mut Position`, obwohl sie nur lesen sollte. Grund: `has_one_move_connect` und Mobilitätsberechnung mutieren `pos.player`.
**Änderung:**
- In `board.rs`: Neue Methode `generate_moves_for(&self, player: u8, out: &mut MoveList)` implementieren. Logik aus `generate_moves` kopieren, aber `self.player` ignorieren und stattdessen den übergebenen `player` nutzen.
- In `board.rs`: `generate_moves` delegiert an `generate_moves_for(self.player, out)`.
- In `evaluate.rs:161`: `has_one_move_connect(pos: &mut Position, ...)` → `has_one_move_connect(pos: &Position, ...)`.
  - Temporäres `pos.player = player` entfernen.
  - Stattdessen `pos.generate_moves_for(player, &mut moves)` nutzen.
  - `make_move`/`unmake_move` entfernen - stattdessen nur Connectivity prüfen? Nein, `has_one_move_connect` macht `make_move` und prüft `is_connected`. Das ist notwendig.
  - Aber: `make_move` mutiert. Daher kann `has_one_move_connect` nicht auf `&Position` arbeiten, wenn es `make_move` nutzt.
  - **Lösung:** `has_one_move_connect` nimmt `&Position` und klont die Position für jeden Testzug. Da nur bis zu 6 Züge geprüft werden, ist das akzeptabel.
  - Implementierung: `let mut test_pos = pos.clone(); test_pos.player = player; if test_pos.make_move(...) { ... }`.
- In `evaluate.rs:307-320` (Mobilität): Analog - `generate_moves_for` nutzen statt `pos.player` zu mutieren.
- In `search.rs`: Alle `evaluate(&mut pos, ...)` → `evaluate(&pos, ...)` ändern.
  - `search.rs:509`, `search.rs:511`, `search.rs:536`, `search.rs:539`, `search.rs:758`.
- In `search.rs:750`: `quiescence(&mut pos, ...)` - hier bleibt `&mut`, da `quiescence` tatsächlich Züge macht.
  - Aber `quiescence` ruft `evaluate` auf: `search.rs:758` - das wird dann `&pos`.
**Verifikation:** `cargo build`, `cargo test`. Keine Compiler-Fehler bei `&mut` → `&`.

- Alternative bessere Lösung: Delta-Pruning nur auf Captures anwenden, nicht auf "connecting moves" (die in Quiescence als noisy eingestuft werden).
  - In `search.rs:802`: Bedingung `if !capture && stand_pat + Q_DELTA < alpha` ändern.
  - Da "connecting moves" kein `capture` sind, würden sie geprunt. Besser: `Q_DELTA` auf `5000` setzen, oder spezifisch für connectivity-Züge ausnehmen.
**Verifikation:** Self-Play: Positionen mit nahem Zusammenschluss testen. Ohne Fix: Engine pruned den Verbindungszug und verliert.

## Task 11: Flood-Fill-Cache für Eval (M)
**Dateien:** `src/board.rs`, `src/evaluate.rs`
**Warum:** 6 Flood-Fills pro Eval sind der Performance-Bottleneck.
**Änderung:**
- In `board.rs:251` (Position): Neue Felder hinzufügen:
  ```rust
  pub cached_largest: [i32; 2],
  pub cached_components: [i32; 2],
  pub cached_spread: [i32; 2],
  pub cache_valid: [bool; 2],
  ```
- In `board.rs:478-554` (make_move): Nach Move, setze `cache_valid[0] = false; cache_valid[1] = false;` (oder nur für den Spieler, dessen Zug es betrifft, aber Captures können auch den Gegner beeinflussen).
  - Einfacher: Beide invalidieren.
- In `board.rs:556-605` (unmake_move): Beide invalidieren.
- In `evaluate.rs:271-287`: Statt direkte `pos.largest_component_value(...)` und `pos.component_count(...)`, zuerst Cache prüfen:
  ```rust
  fn get_largest(pos: &mut Position, player: u8) -> i32 {
      let idx = if player == ONE { 0 } else { 1 };
      if !pos.cache_valid[idx] { recompute_cache(pos, player); }
      pos.cached_largest[idx]
  }
  ```
  - Aber: `evaluate` soll nach Task 5 `&Position` sein. Cache-Invalidierung und -Berechnung erfordert `&mut`.
  - **Lösung:** Cache-Berechnung in `make_move`/`unmake_move` inkrementell? ODER: `evaluate` doch kurzzeitig `&mut` nehmen? Nein, das würde Task 5 rückgängig machen.
  - **Bessere Lösung:** Cache-Werte werden in `make_move`/`unmake_move` inkrementell aktualisiert (nur die betroffenen Komponenten). Das ist komplex, aber sauber.
  - **Alternative:** Eval nimmt `&mut Position` für den Cache-Refresh, aber das ist unsauber.
  - **Empfohlene Alternative:** Task 12 (Inkrementelle Konnektivität) stattdessen implementieren, das macht den Cache überflüssig.
**Verifikation:** Falls implementiert: `cargo test`, Self-Play NPS-Vergleich.

## Task 12: Inkrementelle Konnektivität (L)
**Dateien:** `src/board.rs`, `src/evaluate.rs`
**Warum:** O(Fische) Flood-Fill pro Eval → O(1) statt O(Fische) durch Union-Find oder inkrementelle Komponenten-Tracking.
**Änderung:**
- In `board.rs:251` (Position): Union-Find-Struktur pro Spieler (oder Array von Komponenten-IDs).
  - `component_id: [[u8; 100]; 2]` (0 = nicht der Spieler, 1..N = Komponente).
  - `component_size: [[u8; 100]; 2]` (nur für Root-Indices).
  - `component_value: [[u16; 100]; 2]`.
- In `board.rs:478-554` (make_move):
  - From-Square: Entfernt aus Komponente → evtl. Split?
  - To-Square: Hinzufügen → Union mit Nachbarn.
  - Capture: Entfernt aus Gegner-Komponente.
  - Dies ist NICHT trivial beim Split! Wenn ein Fisch aus der Mitte einer Kette entfernt wird, splittet sich die Komponente potenziell in 2+ Teile.
  - Split-Erkennung erfordert BFS/DFS vom From-Square aus über Nachbarn.
  - Da max. 8 Fische pro Seite, ist ein Split-BFS sehr schnell (max. 7 Nachbarn zu prüfen).
- In `board.rs:556-605` (unmake_move): Noch komplexer - muss vorherigen Zustand restoren.
  - Einfacher: `Undo` speichert vorherige component_id/size/value Arrays. Bei max. 8 Fischen = 100 Bytes pro Array * 3 Arrays = 300 Bytes pro Undo. Akzeptabel.
- In `evaluate.rs`: `largest_component_value`, `component_count`, `component_spread` nutzen inkrementelle Werte statt Flood-Fill.
**Verifikation:** Extensive Unit-Tests (1000 Zufallszüge + Invarianten-Check), Self-Play NPS x3-5 erwartet.
